# GitFlow для проекта цветочного сайта

В проекте используется система контроля версий Git. Репозиторий размещён на платформе GitHub и используется для хранения кода сайта и отслеживания изменений
Для организации процесса разработки применяется упрощённая стратегия ветвления, основанная на GitFlow

## Основные ветки

*main* — основная стабильная версия сайта
Эта ветка содержит код, готовый к публикации и использованию. В неё попадают только проверенные изменения

*develop* — ветка разработки
В этой ветке находятся новые изменения и функции сайта, которые находятся на этапе тестирования перед добавлением в основную версию

## Вспомогательные ветки

Для разработки отдельных функций создаются дополнительные ветки

*feature/* — используется для разработки новой функциональности сайта
Ветка создаётся от develop и после завершения работы объединяется обратно через Pull Request

Примеры веток:
- feature/catalog-page — разработка страницы каталога
= feature/cart-functionality — добавление корзины
= feature/order-form — реализация формы заказа

*bugfix/* — используется для исправления ошибок во время разработки

Примеры:
- bugfix/catalog-images
- bugfix/mobile-layout

*hotfix/* — используется для срочных исправлений в основной версии сайта
Создаётся от ветки main и после исправления объединяется обратно в main и develop

## Правила именования веток

Названия веток должны:
1. использовать нижний регистр
2. содержать краткое описание задачи
3. использовать дефисы для разделения слов

Примеры:
1. feature/product-card
2. feature/delivery-page
3. bugfix/cart-total

## Процесс работы

Процесс разработки в репозитории строится следующим образом:
1. Создаётся новая задача или изменение
2. От ветки develop создаётся новая ветка feature/название
3. Вносятся изменения в код проекта
4. Выполняются коммиты с описанием изменений
5. Ветка отправляется в удалённый репозиторий GitHub
6. Создаётся Pull Request для объединения изменений в develop
7. После проверки изменения объединяются с основной веткой разработки
Периодически изменения из ветки develop объединяются с веткой main для выпуска новой версии сайта

## Коммиты

Для удобства работы используются понятные сообщения коммитов

Рекомендуется использовать следующие префиксы:
- feat: добавление новой функции
- fix: исправление ошибки
- docs: изменения в документации
- style: изменения стилей или верстки
- refactor: переработка кода
- test: добавление тестов

Примеры:
1. feat: добавить страницу каталога
2. fix: исправить отображение изображений
3. tyle: обновить стили карточек товаров

## Тегирование версий

При выпуске новой версии проекта в ветке main создаётся тег версии

Формат версии:

*v<major>.<minor>.<patch>*

Пример:
*v1.0.0*

Используется принцип семантического версионирования, где:
- *major* — крупные изменения проекта
- *minor* — добавление новой функциональности
- *patch* — исправление ошибок
